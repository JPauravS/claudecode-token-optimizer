# Contributing

Thanks for looking. This is a solo-maintainer project; contributions are welcome but review may be slow.

## Before you start

- Open an issue to discuss non-trivial changes before writing code. A 2-minute "here's what I'd do" comment saves both of us an hour of review-rewrite loops.
- Bug reports: attach `npm run doctor` output and the `dashboard/data/diagnostic-*.log` path the installer prints. Those cover 80% of install failures.

## Local setup for contributors

```bash
git clone https://github.com/JPauravS/claudecode-token-optimizer.git
cd claudecode-token-optimizer
npm install
npm run fetch-openwolf          # fetches vendored .ts sources (gitignored)
```

## Running tests

```bash
node hooks/caveman-session-stop.test.js
node scripts/openwolf-prompt.test.js
node scripts/repair-ledger.test.js
for t in scripts/patches/*.test.js; do node "$t"; done
```

All must print `PASS`. CI (`.github/workflows/test.yml`) runs the same on every push.

## PR policy

### Touching our own code
- Small, focused PRs preferred. One logical change per PR.
- Commit messages: conventional prefix (`feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, `test:`).
- Don't rewrite history (`--amend`, force-push) after a review starts.

### Touching vendored subtrees (`hooks/caveman-*` / `hooks/openwolf/` / `skills/caveman*`)
- **Don't hand-edit** — vendored files are regenerated by `npm run fetch-caveman` / `npm run fetch-openwolf` from pinned upstream SHAs.
- For behavioral changes, add or extend a patch in `scripts/patches/<name>.patch.js` with a matching `*.patch.test.js`. The test confirms the patch still applies after the next `fetch-*` sync.
- For upstream SHA bumps, update `scripts/{caveman,openwolf}-sources.json` and confirm all `scripts/patches/*.test.js` still pass after `npm run fetch-openwolf`.

### Adding features
- If it increases the install surface (new hook, new CLI arg, new dashboard tab), document it in `README.md` and add a `ROADMAP.md` entry for decision history.
- Keep the "zero telemetry, 127.0.0.1-only" posture. Any outbound connection needs a very good justification and an opt-in flag.

## Before opening a PR

- [ ] Tests pass locally
- [ ] `npm run doctor` (if applicable) still passes
- [ ] README / ROADMAP / LICENSE-ATTRIBUTION updated if the change touches those surfaces
- [ ] No new absolute paths (`/Users/<you>/…`, `C:\Users\<you>\…`) in committed code
- [ ] No phase / version / internal audit jargon in comments
